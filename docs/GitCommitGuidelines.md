## 概述

本文档规定 Daily 项目的 Git 提交文案标准，旨在提升代码历史的可读性、可维护性和可追溯性。所有开发人员必须严格遵守本规范。

---

## 提交文案格式

每次提交必须采用以下格式：

```text
<type>: <subject>

<body>

<footer>
```

### 标题行（Subject）

- **必填**，长度不超过 50 个字符
- 使用中文编写，表述简洁准确
- 格式为 `<类型>: <描述>`，注意冒号后有一个空格
- 描述使用祈使句，说明本次提交的意图
- 末尾不加句号

### 正文（Body）

- **可选**，当提交涉及多个文件或复杂逻辑时必填
- 每行不超过 72 个字符
- 详细说明修改内容、原因和影响范围
- 使用 `-` 符号列出具体变更项

### 页脚（Footer）

- **可选**，用于引用 Issue 或 Breaking Change 说明
- 格式为 `Closes #123` 或 `BREAKING CHANGE: 说明`

## 类型说明

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| `feat` | 新功能 | 新增用户可见的功能或特性 |
| `fix` | 修复 | 修复代码缺陷或安全问题 |
| `docs` | 文档 | 仅修改文档内容（README、API 文档等） |
| `style` | 格式 | 代码格式调整（缩进、空格、分号等），不影响功能 |
| `refactor` | 重构 | 代码重构，既不修复缺陷也不新增功能 |
| `perf` | 性能 | 提升性能的代码优化 |
| `test` | 测试 | 新增或修改测试代码 |
| `chore` | 构建 | 构建工具、依赖更新、配置调整等 |
| `ci` | 持续集成 | CI/CD 配置文件修改 |
| `revert` | 回滚 | 撤销之前的提交 |

## 正文编写规范

### 必须遵循的原则

1. **单一职责**：每次提交只做一件事，避免大范围混合提交
2. **清晰描述**：说明"做了什么"和"为什么做"
3. **影响范围**：明确指出修改涉及的模块或组件

### 推荐的正文结构

```text
- 修改内容：具体说明变更点
- 修改原因：说明为什么需要这次变更
- 影响范围：列出受影响的文件或模块
- 测试情况：说明是否经过测试及测试结果
```

## 审核清单

提交代码前，请逐项确认：

- [ ] 标题行是否使用正确的类型前缀？
- [ ] 标题行是否控制在 50 字符以内？
- [ ] 标题行是否使用祈使句表述？
- [ ] 正文是否详细说明修改内容和原因？
- [ ] 是否避免一次提交包含不相关的多个变更？
- [ ] 提交内容是否与标题描述一致？
- [ ] 是否检查过拼写和语法错误？

## 示例参考

### 示例 1：新增功能

```text
feat: 新增用户积分兑换功能

- 新增积分兑换接口 /api/points/exchange
- 新增兑换记录表 exchange_records
- 新增兑换规则配置项
- 前端新增兑换页面和兑换历史组件

影响范围：后端积分服务、前端积分中心页面
```

### 示例 2：修复缺陷

```text
fix: 修复抽奖并发超卖问题

- 抽奖流程添加数据库事务保护
- 库存扣减使用条件更新实现乐观锁
- 积分扣减添加原子性校验

修复原因：并发场景下多个用户同时抽奖可能导致库存为负数
测试情况：新增并发测试用例，100 次并发请求全部通过
```

### 示例 3：文档更新

```text
docs: 更新 API 接口文档和部署指南

- 补充管理后台接口文档
- 更新环境变量配置说明
- 新增 PM2 部署最佳实践

影响范围：docs/TechnicalDocumentation.md、README.md
```

### 示例 4：性能优化

```text
perf: 优化连续打卡计算算法

- 将逐天查询改为单次批量查询
- 使用 Set 数据结构优化日期查找效率
- 时间复杂度从 O(n) 降至 O(1)

性能对比：100 次调用从 1000ms 降至 10ms
```

### 示例 5：代码重构

```text
refactor: 提取分页逻辑为独立工具函数

- 新增 paginationHelper.js 统一分页查询
- 替换用户服务、积分服务、打卡服务中的重复分页代码
- 新增分页工具函数单元测试

影响范围：server/src/services/ 目录下多个服务文件
```

## 常见错误

### 错误 1：标题过于笼统

```text
❌ fix: 修复一些问题
❌ feat: 新增功能
```

### 错误 2：范围过大的混合提交

```text
❌ fix: 修复多个模块的缺陷并优化性能
（应该拆分为多个独立提交）
```

### 错误 3：类型使用不当

```text
❌ feat: 修复登录接口的 SQL 注入问题
（修复缺陷应使用 fix 而非 feat）
```

### 错误 4：缺少正文说明

```text
❌ fix: 修复抽奖问题
（复杂修复应补充正文说明具体修复内容）
```
